= Troubleshooting the BeeGFS CSI Driver Deployment
:hardbreaks:
:nofooter:
:icons: font
:linkattrs:
:imagesdir: ../media/


[.lead]
In guidance for troubleshooting issues with the BeeGFS CSI Driver deployment, ensure sure that:

* All pre-requisites have been, including BeeGFS client setup and configuration. 
* The BeeGFS-CSI driver has been deployed using the appropriate overlays. 
* Verify the deployment and troubleshoot any errors, such as taint issues or swap being enabled. 
* Example applications have been deployed and validated to confirm functionality.

. *Troubleshooting BeeGFS Client and Helperd Configuration*

Review the configuration in /etc/beegfs/beegfs-client.conf. 

* If your BeeGFS file system should use connection authentication, and to do that ensure the following flag is set to false: 

[source,console]
----
connAuthFileEnabled = false
---- 
NOTE: If the customer is willing for the BeeGFS file system to connect without 	authentication, then the flag above should be set to true.

* Update the IP address for the management service in the sysMgmtdHost parameter. 
* Update the path for the connRDMAInterfacesFile and connInterfacesFile. This file contains the name of the interfaces used for storage or inter-node communications. Eg.
[source,shell]
----
ibs1f0 
ibs1f1 
----
* Update the path for the connAuthFile parameter if authentication is enabled. 
Example configuration screenshot shown below:
[source,shell]
----
connDisableAuthentication = false 
# BEGIN ANSIBLE NETAPP_ESERIES.HOST.UPDATE_CONF MANAGED BLOCK

connMaxConcurrentAttempts=4  
connRDMABufNum=36  
connRDMABufSize=65536  
tuneFileCacheType=native  
tuneFileCacheBufSize=2097152  
connFallbackExpirationSecs=90  
connCommRetrySecs=600  
sysSessionChecksEnabled=False connRDMAInterfacesFile=/etc/beegfs/XXX.X.XX.X_8004_connInterfaces.conf sysMountSanityCheckMS=11000

connInterfacesFile=/etc/beegfs/XXX.X.XX.X_8004_connInterfaces.conf connAuthFile=/etc/beegfs/XXX.X.XX.X connAuthFile 
----
* Review the configuration in /etc/beegfs/beegfs-helperd.conf.
** If your BeeGFS file system uses connection authentication, ensure the following flag is set to false in the helperd conf file as well: 
[source,console]
----
connDisableAuthentication = false 
----

NOTE: If the customer is willing for the BeeGFS file system to connect without 	authentication, then the flag above should be set to true.

[source,shell]

[start=5]
. *Troubleshooting BeeGFS Controller issues*

After deploying overlays, we may find status as PENDING in the output of kubectl get all. 
[source,console]
----
root@node01:/home/user_id/beegfs-csi-driver# kubectl get all -n beegfs-csi 
NAME                          READY   STATUS    RESTARTS   AGE 
pod/csi-beegfs-controller-0   0/3     Pending   0          59s 
----

If the status is as shown above, then it is time to explore the pods created by using the describe command. 
[source,shell]
----
kubectl describe pod csi-beegfs-controller-0 -n beegfs-csi 
----
If you encounter errors related to "untolerated taint" (visible in the output of kubectl describe) as below: 
[source,console]
----
Events: 
  Type     Reason            Age   From               Message 
  ----     ------            ----  ----               ------- 
  Warning  FailedScheduling  84s   default-scheduler  0/1 nodes are available: 1 node(s) had untolerated taint {node.kubernetes.io/disk-pressure: }. preemption: 0/1 nodes are available: 1 Preemption is not helpful for scheduling. 
----
Remove the taint from the node using the command: 
[source,shell]
----
kubectl taint nodes node01 node-role.kubernetes.io/control-plane:NoSchedule-
----
Expected Output:
[source,console]
----
root@node01:/home/user_id/beegfs-csi-driver# kubectl taint nodes node01 node-role.kubernetes.io/control-plane:NoSchedule-  
error: taint "node-role.kubernetes.io/control-plane:NoSchedule" not found 
----

After removing the taint, re-apply the overlays. The issue will then be resolved: 
[source,shell]
----
kubectl apply -k deploy/k8s/overlays/default 
----
