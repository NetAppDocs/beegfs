Guidance for troubleshooting issues with the BeeGFS CSI Driver deployment.


=== Kubernetes setup - Common Error Scenarios 

Error example when trying to get all the currently running pods on the node: 

[.command-window]   
----
kubectl get pods
----
.Example of Erroneous Output: 
image:CSI_KB_12.png["Screenshot showing error retrieving Kubernetes pods"]

There are several areas of investigation for fixing most of the issues like these. We can look at the areas in this order: 
[.indented]
====

*Error in containerd*

1. Check the status of the containerd daemon: 
[.command-window] 
systemctl status containerd
.Expected Output:
image:CSI_KB_13.png["Screenshot showing containerd service active and running"]

2. If the daemon is not running, restart it:
[.command-window] 
systemctl restart containerd
.Expected Output:
image:CSI_KB_14.png["Screenshot showing containerd service active and running"]

[.indented]
====


[.indented]
====

*Error in kubelet*

1. Check the status of the kubelet service: 
[.command-window]
systemctl status kubelet
.Expected Output:
image:CSI_KB_15.png["Screenshot showing kubelet service active and running"]

2. If the service is not running, restart it:
[.command-window]
systemctl restart kubelet
.Expected Output:
image:CSI_KB_16.png["Screenshot showing kubelet service active and running"]

If issue persists, check the syslog for errors:
[.command-window]
tail -f /var/log/syslog | grep kubelet   

*Swap Issue*
If you see errors related to "swap", disable swap and restart kubelet(NOTE: For the below command for `swapoff`, I suggest typing it and not copying it from here. You might face errors if the hyphen changes from a standard ASCII hyphen-minus (-) to a Unicode en dash (–)): 
[.command-window]
swapoff -a
[.command-window]
systemctl restart kubelet

.Expected Output:
image:CSI_KB_17.png["Screenshot showing kubelet service active and running"]

Note: Swap is typically enabled by default during OS installation and listed in /etc/fstab. Kubernetes does not support running with swap enabled. 

[.indented]
====



=== Troubleshooting BeeGFS Client and Helperd Configuration

[.indented]
====

Review the configuration in /etc/beegfs/beegfs-client.conf. 

* If your BeeGFS file system should use connection authentication, and to do that ensure the following flag is set to false: 

[source,yaml]
connAuthFileEnabled = false
 
NOTE: If the customer is willing for the BeeGFS file system to connect without 	authentication, then the flag above should be set to true.

* Update the IP address for the management service in the sysMgmtdHost parameter. 
* Update the path for the connRDMAInterfacesFile and connInterfacesFile. This file contains the name of the interfaces used for storage or inter-node communications. Eg.
image:CSI_KB_18.png["Screenshot showing interfaces"]
* Update the path for the connAuthFile parameter if authentication is enabled. This file contains the authentication keys required for secure connections.
Example configuration screenshot shown below:
image:CSI_KB_19.png["Screenshot showing connAuthFile path configuration"]
* Review the configuration in /etc/beegfs/beegfs-helperd.conf.
* If your BeeGFS file system uses connection authentication, ensure the following flag is set to false in the helperd conf file as well: 
[source,yaml]
connDisableAuthentication = false 

NOTE: If the customer is willing for the BeeGFS file system to connect without 	authentication, then the flag above should be set to true. 

Example configuration screenshot shown below: 
image:CSI_KB_20.png["Screenshot showing connDisableAuthentication configuration"]

[.indented]
====


=== Troubleshooting Airgapped Kubernetes Deployment 


[.indented]
====
* If your air-gapped environment does not have a Docker Hub mirror, then you pull the necessary images from a machine with access to the internet then save them as tar files.  

* List of docker images can be found in the yaml files in deploy/k8s/overlays/default/kustomization.yaml.

image:CSI_KB_21.png["Screenshot showing docker images in kustomization.yaml"]

* From a machine with access to the internet, save them as tar files as shown below: 
[.command-window]
docker pull docker.io/netapp/beegfs-csi-driver 
[.command-window]
docker save docker.io/netapp/beegfs-csi-driver > beegfs-csi-driver.tar

* Copy the tar file to the airgapped kubernetes node and load them as shown below: 
[.command-window]
docker load < beegfs-csi-driver.tar.gz

Once the images are available, copy deploy/k8s/overlays/default to deploy/k8s/overlays/my-overlay and modify the images to point to the images now locally available. Adjust the images[].newTag fields as necessary to ensure they either match images that exist on the Kubernetes nodes or reference your internal registry. 

[.indented] 

====


=== Troubleshooting BeeGFS Controller issues


[.indented] 
====
After deploying overlays, we may find status as PENDING in the output of kubectl get all.  
image:CSI_KB_22.png["Screenshot showing docker images in kustomization.yaml"]

If the status is as shown above, then it is time to explore the pods created by using the describe command. 
[.command-window]
kubectl describe pod csi-beegfs-controller-0 -n beegfs-csi 

If you encounter errors related to "untolerated taint" (visible in the output of kubectl describe), remove the taint from the node using the command shown below: 

image:CSI_KB_23.png["Screenshot showing docker images in kustomization.yaml"]
[.command-window]
kubectl taint nodes node01 node-role.kubernetes.io/control-plane:NoSchedule-

Expected Output:
image:CSI_KB_24.png["Screenshot showing docker images in kustomization.yaml"]

After removing the taint, re-apply the overlays. The issue would be resolved: 
[.command-window]
kubectl apply -k deploy/k8s/overlays/default 

[.indented]
